  ── Tour 1
  Utilisateur : Explique-moi ce qu'est un pipeline RAG et comment il fonctionne de bout en bout.

  Métriques :
    Citation : Oui  |  Structure : OK  |  Docs : 4  |  Ancrage : 0.75  |  Réf. hist. : Non

  ResearchPal :
     ---
    **Réponse**
    Un pipeline RAG (Retrieval-Augmented Generation) est une architecture qui améliore les modèles de langage (LLM) en leur permettant d'accéder à des informations externes au moment de la génération. Il fonctionne en quatre étapes :
    1. Ingestion [1] : Les documents source sont découpés en chunks, vectorisés avec un modèle d'embeddings, puis stockés dans une base vectorielle.
    2. Retrieval [1] : Lorsqu'un utilisateur pose une question, elle est également vectorisée. Les chunks les plus proches dans l'espace vectoriel sont récupérés.
    3. Augmentation [1] : Les chunks récupérés sont insérés dans le contexte du prompt envoyé au LLM, accompagnés de la question de l'utilisateur.
    4. Génération [1] : Le LLM génère une réponse en s'appuyant sur les chunks fournis.
    
    Il existe plusieurs stratégies pour récupérer les documents pertinents telles que `search_type="similarity"` [3] et `search_type="mmr"` [3]. Le `MultiQueryRetriever` encapsule un retriever existant et génère automatiquement plusieurs variantes de la question pour maximiser le rappel.
    
    Pour maintenir le contexte entre les tours de conversation, LangChain propose différentes options comme `ChatMessageHistory`, `ConversationBufferMemory`, et `ConversationSummaryMemory` [4]. Dans un pipeline RAG avec LCEL, l'historique est passé via `MessagesPlaceholder` dans le prompt template.
    
    En outre, ChromaDB est une base de données vectorielle open-source optimisée pour le RAG qui permet de stocker des vecteurs avec leurs métadonnées associées, d'effectuer des recherches par similarité de façon efficace, et de persister les données localement sur disque [4].

  ── Tour 2
  Utilisateur : Comment la stratégie MMR améliore-t-elle les résultats par rapport à la similarité cosinus ?

  Métriques :
    Citation : Oui  |  Structure : OK  |  Docs : 4  |  Ancrage : 0.56  |  Réf. hist. : Non

  ResearchPal :
     ---
    **Réponse**
    La stratégie MMR (Maximal Marginal Relevance) a pour objectif d'améliorer les résultats en comparaison de la similarité cosinus. Elle permet d'éviter le retour de chunks trop similaires entre eux en combinant pertinence et diversité. Cela est utile lorsque les documents contiennent beaucoup de répétitions ([2](text) intro_rag.txt, [3](text) intro_rag.txt). En utilisant la stratégie MMR plutôt que la similarité cosinus seul peut améliorer le rappel pour les questions ambiguës ou complexes.

    Ce contexte n'indique pas explicitement comment la stratégie MMR améliore le rappel par rapport à la similarité cosinus, mais il suggère que cela est possible en combinant pertinence et diversité.

  ── Tour 3
  Utilisateur : En tenant compte de tout ce que tu viens d'expliquer, quelle stratégie de retrieval recommanderais-tu pour un corpus académique avec beaucoup de documents redondants ?

  Métriques :
    Citation : Oui  |  Structure : OK  |  Docs : 4  |  Ancrage : 0.77  |  Réf. hist. : Non

  ResearchPal :
     ---
    **Réponse**
    Pour un corpus académique contenant de nombreux documents redondants, je recommanderais la stratégie MMR (Maximal Marginal Relevance) plutôt que la similarité cosinus. MMR combine la pertinence et la diversité pour éviter de retourner des chunks trop similaires entre eux [1]. Cela est utile lorsque les documents contiennent beaucoup de répétitions.
    Toutefois, il est important d'optimiser le pipeline RAG pour améliorer les performances du système. Utiliser une stratégie comme `MultiQueryRetriever` qui génère plusieurs reformulations de la question peut également aider à maximiser le rappel en utilisant diverses formulations [4].
